面试可能问到的问题及参考答案

1) 你如何用 Tailwind CSS 搭建并维护一套可扩展的样式体系？

参考答案：
我会先在 `tailwind.config` 里统一设计令牌（颜色、间距、圆角、阴影、字号和断点），再约定组件层的语义化 class 组合方式，避免页面里出现大量随意拼接。对于高复用样式，我会封装成组件或抽象为可复用的 class 组合，减少重复。多人协作时会补充 lint 规则和代码评审清单，保证命名和使用一致。这样做的好处是可维护性高，后续改主题或改规范时成本可控。

1) React 和 Vue 都写过的话，你对其中一个“深入理解”体现在哪？

参考答案：
我对 React 更深入，重点是组件状态管理、渲染机制和性能优化。比如我会明确区分“状态归属”和“数据流向”，避免不必要的状态提升；在性能上结合 `memo`、`useMemo`、`useCallback` 做针对性优化，而不是滥用。复杂页面会按“容器组件 + 展示组件”拆分，并把副作用逻辑沉淀到自定义 Hook，提升可测试性和复用性。

1) 你会如何实现一个多终端响应式页面？

参考答案：
我通常采用“移动优先 + 渐进增强”的方式，先保证小屏核心流程，再按断点扩展平板和桌面布局。布局层面优先用 Flex/Grid，组件层面处理不同终端的交互差异（点击区域、手势、悬浮态）。同时会在主流设备和浏览器上做回归验证，关注字体缩放、横竖屏切换、容器溢出等细节，确保体验一致。

1) 你做过哪些前端性能优化？怎么衡量效果？

参考答案：
我会从“首屏加载、运行时渲染、资源体积”三层优化。首屏阶段做路由级懒加载、静态资源压缩和缓存策略；渲染阶段减少无效重渲染、控制长列表渲染成本；资源层面做按需加载和图片优化。衡量上会看 LCP、CLS、INP/交互延迟、包体积和关键页面耗时，并在优化前后保留对比数据，避免只凭主观感觉判断。

1) 在 Web3 项目里，你如何处理钱包连接和链上交互？

参考答案：
我会先抽象“钱包适配层”，统一连接、断开、账户切换、链切换和错误处理，避免业务代码里到处写钱包细节。合约交互会区分“读操作”和“写操作”：读走稳定的查询流程，写操作明确展示 pending/success/fail 状态并给出交易哈希。对于常见异常（用户拒签、余额不足、网络错误、gas 估算失败）会做可理解的提示和兜底引导，保证用户可恢复。

1) 如果后端接口慢或不稳定，你在前端怎么保障体验？

参考答案：
我会结合骨架屏、加载状态分级、超时重试和错误降级策略。关键路径接口优先保障并做超时提示，非关键模块可延后加载。对于可缓存的数据会做本地缓存和失效策略，减少重复请求。交互上给用户明确反馈（加载中、失败原因、可重试入口），让页面“慢但可用”，而不是“卡住不可用”。

1) 说一个你和产品/设计/后端协作推进需求落地的例子。

参考答案：
我通常会在需求评审阶段先识别风险点（边界状态、接口依赖、交互冲突），并把问题前置到文档和评审会上。开发中用可视化 Demo 和接口 mock 提前对齐，减少联调返工。上线前和测试一起梳理核心回归清单，确保关键路径稳定。我的目标是让沟通有结论、有 owner、有截止时间，推动问题闭环。

1) 你如何保证代码规范和可维护性？

参考答案：
我会通过 ESLint + Prettier + TypeScript（如项目支持）建立统一规范，再通过目录分层和模块边界控制复杂度。组件设计上遵循单一职责，高复用逻辑抽离到 hooks/utils，避免“巨石组件”。同时在 PR 里关注可读性和可演进性，而不只是“功能能跑”，确保后续迭代不被历史包袱拖慢。

1) 你在 Vite / Webpack 工程化方面做过哪些实践？

参考答案：
我更常用 Vite，实践包括按环境管理配置、代理和构建参数分离、按路由拆包、构建产物分析和缓存策略优化。Webpack 项目中也做过 alias 管理、loader/plugin 精简和构建耗时治理。核心思路是让本地开发体验快、线上包体合理、发布过程可观测可回滚。

1) 你为什么适合这个岗位？

参考答案：
这个岗位强调 Tailwind + React/Vue + 多端适配 + Web3 实战，这和我的技术栈匹配度很高。我不仅能完成业务开发，也能在样式体系、组件抽象和性能优化上给团队带来长期收益。协作上我习惯主动推进问题闭环，能在跨角色配合中稳定交付结果。
