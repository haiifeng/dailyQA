# 一面：原理、工程实践与现场分析

## 1. JavaScript 运行机制

### Q1：浏览器中的一段 JavaScript 代码从下载到执行经历了什么？

回答要点：

- HTML 解析器遇到普通 `script` 时通常暂停解析，下载并执行脚本；`defer` 在 HTML 解析完成后按顺序执行；`async` 下载完成后立即执行，不保证顺序。
- JavaScript 引擎会解析源码、生成语法结构和可执行代码，并通过解释执行与即时编译优化热点代码。
- 执行过程涉及调用栈、词法环境、堆内存和垃圾回收。
- 脚本还可能触发样式计算、布局和绘制，但 JavaScript 执行与页面渲染不会在同一时刻并行修改页面。
- 实际优化时要同时关注脚本体积、执行时间和主线程长任务，不能只看下载速度。

### Q2：事件循环中宏任务、微任务和页面渲染的关系是什么？

回答要点：

- 浏览器从任务队列取出一个任务执行，例如脚本、定时器或事件回调。
- 当前任务结束并清空调用栈后，会依次清空微任务队列，例如 `Promise.then` 和 `queueMicrotask`。
- 微任务持续产生微任务可能阻塞渲染，不能认为微任务一定更高效。
- 浏览器在合适的时机执行渲染；`requestAnimationFrame` 回调通常在下一次绘制前运行。
- Node.js 的事件循环阶段与浏览器不同，回答时要先说明运行环境。

### Q3：闭包为什么可能导致内存问题？如何判断它是不是泄漏？

回答要点：

- 闭包会保留对外层词法环境中变量的引用，只要闭包可达，相关对象就可能无法被回收。
- 闭包本身不是泄漏；缓存无上限、事件监听未解绑、定时器未清理或 DOM 被意外引用才是常见原因。
- 使用 Chrome Memory 面板对比多次 Heap Snapshot，观察对象数量和 Retained Size 是否持续增长。
- 通过引用链找到 GC Root，定位是谁仍然持有对象。
- 修复后重复相同操作，确认内存能够回落，而不是只凭代码观察下结论。

### Q4：什么情况下会触发 V8 的隐藏类退化？业务中需要刻意优化吗？

回答要点：

- 对象以不同顺序添加属性，或频繁改变对象结构，可能产生不同隐藏类，降低内联缓存命中率。
- 对热点路径中的大量同类对象，保持稳定结构和一致的属性初始化顺序更友好。
- 普通业务代码不应为了理论优化牺牲可读性。
- 应先用性能分析确认瓶颈确实在对象访问或计算热点，再做针对性调整。

## 2. 浏览器、网络与安全

### Q5：用户输入 URL 后，如何定位“页面打开慢”发生在哪个阶段？

回答要点：

- 先按 DNS、连接建立、TLS、请求等待、内容下载、解析、执行和渲染拆分链路。
- 使用 Navigation Timing、Resource Timing 和 DevTools Network 查看 TTFB、资源瀑布图与连接复用情况。
- TTFB 高要继续区分网关、服务端计算、数据库和第三方依赖。
- 下载快但页面可用慢，要检查 JavaScript 长任务、关键资源阻塞、接口瀑布和 hydration 成本。
- 用真实用户监控按地区、设备、网络和版本分组，避免只依赖开发机数据。

### Q6：如果线上偶发白屏，但本地无法复现，你会怎样排查？

回答要点：

1. 确认影响范围：版本、浏览器、设备、地区、入口和用户比例。
2. 检查静态资源加载失败、Chunk 版本不一致、运行时异常、接口阻塞和根节点渲染失败。
3. 通过全局错误、Promise 拒绝、资源错误和路由信息还原现场。
4. 为动态资源加载失败增加有限重试或版本刷新提示，不能无限刷新。
5. 使用 Error Boundary 提供降级页面，但它不能捕获所有异步和事件回调错误。
6. 结合发布记录、Source Map 和错误聚类定位首个异常版本，并准备快速回滚。

### Q7：同源策略保护了什么？CORS 为什么不能防止 CSRF？

回答要点：

- 同源策略主要限制页面读取其他源的数据，并限制部分跨源 DOM 和存储访问。
- CORS 是服务端授权浏览器允许某些跨源页面读取响应的机制。
- CSRF 利用浏览器自动携带 Cookie 发起请求，攻击者不一定需要读取响应，所以仅配置 CORS 不足以防御。
- CSRF 可通过 `SameSite` Cookie、CSRF Token、Origin/Referer 校验和敏感操作二次确认防护。
- XSS 与 CSRF 的攻击入口不同；XSS 可能直接绕过页面内的很多防护。

### Q8：前端如何降低 XSS 风险？

回答要点：

- 默认使用框架的文本转义能力，避免直接拼接 HTML。
- 必须渲染富文本时使用成熟的白名单净化方案，并限制协议、标签和属性。
- 避免执行字符串形式的代码，不把不可信数据插入脚本、样式或 URL 危险位置。
- 使用 CSP 降低注入代码的执行能力，但 CSP 是纵深防御，不替代输入输出处理。
- 会话 Cookie 使用 `HttpOnly` 和 `Secure`，减少脚本直接读取敏感令牌的机会。

## 3. React 与前端架构

### Q9：为什么 React 状态更新后，不应该立即读取 DOM 判断最终结果？

回答要点：

- 状态更新表达的是下一次渲染意图，React 可能批量处理并安排渲染。
- Render 阶段计算 UI，可被中断或重新执行；Commit 阶段才真正更新 DOM。
- 依赖更新后 DOM 测量的逻辑通常放在 `useLayoutEffect`，非阻塞副作用优先使用 `useEffect`。
- 如果业务逻辑依赖“更新后立即取值”，应优先使用事件参数或计算出的新值，而不是依赖闭包中的旧状态。

### Q10：如何设计一套能同时支持 REST 初始数据和 WebSocket 增量消息的数据层？

回答要点：

- REST 返回带版本或序列号的快照，WebSocket 返回带主键、版本和事件类型的增量消息。
- 建立统一规范化 Store，以业务 ID 存储实体，避免维护多份互相冲突的数据。
- 先建立 WebSocket 缓冲，再加载快照，按序列号应用快照之后的消息，减少窗口期丢数据。
- 对重复、乱序和缺口消息做检测；发现缺口时重新请求快照。
- UI 订阅最小数据切片，并批量合并高频消息，避免每条行情都触发整页渲染。

### Q11：大型前端项目什么时候适合拆微前端？

回答要点：

- 首先看组织边界：多个团队是否需要独立开发、发布和故障隔离。
- 单纯因为代码量大，不足以构成拆微前端的理由；模块化单体通常成本更低。
- 微前端会引入运行时集成、依赖重复、样式隔离、路由协调、跨应用通信和统一体验等成本。
- 如果采用，应定义清晰的业务边界、共享契约、版本策略、监控和降级方式。
- 可先从独立部署的垂直页面开始，而不是一开始拆到组件粒度。

### Q12：设计组件库时，怎样避免“万能组件”失控？

回答要点：

- 组件库解决稳定、重复的交互和视觉问题，不承载具体业务流程。
- 先提供少量清晰的基础能力，真实需求出现后再扩展，不为未知场景堆积布尔属性。
- 复杂结构使用组合和插槽，数据行为通过明确事件暴露。
- 统一可访问性、主题、错误状态和测试约定。
- 通过废弃周期和迁移说明管理破坏性变化，而不是永久兼容所有历史 API。

## 4. AI Agent 与产品工程

### Q13：普通聊天应用和 Agent 产品的核心区别是什么？

回答要点：

- 普通聊天通常是输入到输出；Agent 会围绕目标进行规划、调用工具、观察结果并继续决策。
- Agent 需要管理工具权限、上下文、执行状态、失败恢复和停止条件。
- 产品层要让用户看见当前步骤、关键决策和需要确认的高风险操作。
- 评价不能只看回答是否流畅，还要看任务成功率、调用成本、耗时和安全性。

### Q14：如何防止 Agent 重复调用工具或陷入死循环？

回答要点：

- 限制总步数、总耗时、Token 和工具调用预算。
- 为有副作用的工具定义幂等键，并记录调用参数和结果。
- 检测连续相同动作、状态无变化和重复错误，达到阈值后终止或请求用户介入。
- 工具返回结构化错误，区分可重试、需换方案和不可恢复错误。
- 对转账、删除和发送消息等操作在执行前进行明确确认。

### Q15：RAG 检索结果相关，但最终回答仍然错误，怎样排查？

回答要点：

- 先拆分评估：文档是否正确、切分是否合理、召回是否命中、排序是否正确、生成是否忠于证据。
- 查看命中文本是否包含答案，而不是只看向量相似度分数。
- 混合关键词与向量检索，对候选结果重排，并根据业务元数据过滤权限和时效。
- 要求模型基于证据回答并提供引用；证据不足时明确拒答或请求补充。
- 建立包含标准问题、期望证据和答案要点的评测集，避免靠主观体验调参。

### Q16：Skill、MCP 和普通函数调用分别解决什么问题？

回答要点：

- 普通函数调用描述模型可调用的单个能力及其参数契约。
- MCP 提供统一的协议，让模型客户端发现和调用外部工具、资源或提示模板。
- Skill 更偏向可复用的工作方法，包含触发条件、步骤、约束、参考资料和脚本。
- 三者可以组合：Skill 规定怎样完成任务，MCP 提供外部连接，函数调用执行具体动作。
- 设计时应保持权限最小化，并为有副作用的调用加入确认、审计和错误处理。

## 5. 测试、监控与发布

### Q17：前端测试应该怎样分层？

回答要点：

- 纯函数、状态转换和金额计算使用单元测试，运行快且定位准确。
- 组件测试验证用户交互和关键状态，尽量从用户可见行为断言。
- 少量端到端测试覆盖登录、支付、下单等核心链路，避免把所有分支都堆到 E2E。
- 接口契约测试减少前后端字段或语义变化造成的问题。
- 测试数量不是目标，应优先覆盖错误后果严重、变化频繁和人工回归成本高的部分。

### Q18：如何设计一次低风险的前端发布？

回答要点：

- 构建产物使用内容哈希并长期缓存，HTML 不长期缓存，避免新旧资源错配。
- 使用 Feature Flag 或灰度比例逐步开放功能。
- 发布前验证核心路径、监控面板、告警和回滚方案。
- 发布后关注错误率、白屏率、接口失败、性能指标和业务转化。
- 数据结构或接口变更要保证前后兼容，回滚前端时不能依赖已经不可逆的新数据格式。

### Q19：Source Map 应该公开部署吗？

回答要点：

- Source Map 能把压缩后的错误还原到源码位置，对线上排障很重要。
- 不一定要让浏览器公开访问；可以上传到错误监控平台，并限制访问权限。
- 构建版本、Source Map 和发布记录必须一一对应。
- Source Map 不是敏感信息保护机制，前端代码本身就会下发给用户；真正的密钥不能放在前端。

### Q20：如何定义前端稳定性指标？

回答要点：

- 技术指标：白屏率、JavaScript 错误率、资源加载失败率、接口失败率和关键 Web Vitals。
- 业务指标：登录、支付、下单等关键流程成功率和完成耗时。
- 每个指标要有明确分母、采样规则、版本和环境维度。
- 告警应关注用户影响和异常变化，避免固定阈值产生大量噪声。
- 建立错误发现、归因、止损和复盘闭环，指标本身不能替代行动。

# 一面编码题：数据结构与异步控制

## 题目 1：实现并发受限的异步任务调度器

要求：实现 `runWithLimit(tasks, limit)`，最多同时执行 `limit` 个任务，结果顺序与输入顺序一致；某个任务失败时整体拒绝。

```ts
type Task<T> = () => Promise<T>;

async function runWithLimit<T>(tasks: Task<T>[], limit: number) {
  if (!Number.isInteger(limit) || limit < 1) {
    throw new RangeError("limit must be a positive integer");
  }

  const results = new Array<T>(tasks.length);
  let nextIndex = 0;

  async function worker() {
    while (true) {
      const index = nextIndex++;
      if (index >= tasks.length) return;
      results[index] = await tasks[index]();
    }
  }

  const workerCount = Math.min(limit, tasks.length);
  await Promise.all(Array.from({ length: workerCount }, () => worker()));
  return results;
}
```

- 如果要收集全部错误，而不是首次失败就拒绝，如何修改？
- 如何加入取消能力？

## 题目 2：实现 LRU 缓存

要求：`get` 和 `put` 的平均时间复杂度均为 O(1)。

```ts
class LRUCache<K, V> {
  private readonly cache = new Map<K, V>();

  constructor(private readonly capacity: number) {
    if (!Number.isInteger(capacity) || capacity < 1) {
      throw new RangeError("capacity must be a positive integer");
    }
  }

  get(key: K): V | undefined {
    if (!this.cache.has(key)) return undefined;
    const value = this.cache.get(key)!;
    this.cache.delete(key);
    this.cache.set(key, value);
    return value;
  }

  put(key: K, value: V) {
    this.cache.delete(key);
    this.cache.set(key, value);

    if (this.cache.size > this.capacity) {
      const oldestKey = this.cache.keys().next().value as K;
      this.cache.delete(oldestKey);
    }
  }
}
```

追问：

- 为什么 JavaScript 的 `Map` 可以简化双向链表实现？
- 如果缓存值可能是 `undefined`，怎样区分未命中？
- 如何支持过期时间？

## 题目 3：合并重叠区间

```ts
function mergeIntervals(intervals: Array<[number, number]>) {
  if (intervals.length < 2) return intervals;

  const sorted = intervals
    .map(([start, end]) => [start, end] as [number, number])
    .sort((a, b) => a[0] - b[0]);

  const result: Array<[number, number]> = [sorted[0]];

  for (let i = 1; i < sorted.length; i += 1) {
    const current = sorted[i];
    const last = result[result.length - 1];

    if (current[0] <= last[1]) {
      last[1] = Math.max(last[1], current[1]);
    } else {
      result.push(current);
    }
  }

  return result;
}
```

复杂度：排序时间复杂度 O(n log n)，合并扫描 O(n)，额外空间取决于是否允许修改输入。

# 二面：系统设计与业务决策

## 6. 系统设计

### Q21：设计一个支持多窗口同步的交易下单页面。

回答框架：

1. 服务端是订单状态的最终事实来源。
2. 页面启动时拉取订单快照，并建立 WebSocket 接收增量状态。
3. 同一浏览器窗口间使用 `BroadcastChannel` 传播下单意图和状态变化，减少重复请求。
4. 服务端用客户端请求 ID 保证下单幂等，不能只依赖前端锁。
5. 对账户、交易对和网络变化建立明确的订阅切换与清理流程。
6. 断线重连后重新拉快照，根据版本号修正本地状态。
7. 监控重复下单、状态延迟、消息缺口和下单失败率。

### Q22：设计一个可恢复的文件上传系统。

回答要点：

- 客户端计算文件标识，请求服务端创建上传会话并获取已完成分片。
- 文件分片后限制并发上传，每个分片支持有限重试和校验。
- 暂停或刷新页面后，通过上传会话恢复未完成分片。
- 服务端完成合并后再次校验完整性，并异步进行病毒扫描或内容处理。
- 前端区分排队、上传中、暂停、失败、处理中和完成状态。
- 文件类型和大小必须由服务端再次校验，不能信任前端声明。

### Q23：支付结果页显示成功，但用户余额没有变化，如何处理？

回答要点：

- 前端跳转成功不等于支付最终成功，应查询服务端订单状态。
- 服务端以支付渠道回调或链上确认作为最终依据，并保证回调幂等。
- 支付、入账可能属于不同步骤，使用处理中状态而不是直接显示失败。
- 建立订单号、支付流水和账务流水之间的追踪关系，支持对账和补偿。
- 超过预期时间仍未入账时提供查询入口和客服所需信息，同时触发告警。

## 7. 场景与行为问题

### Q24：产品要求三天上线，但你认为完整方案至少需要两周，怎么办？

建议回答：

- 先确认三天上线背后的业务目标和不能延期的原因。
- 拆出最小闭环、风险项和可延后部分，而不是只给出“做不了”的结论。
- 明确三天方案的范围、人工兜底、监控和已接受风险，并让相关负责人共同确认。
- 对资金、安全和合规风险不通过隐藏质量问题换速度。
- 上线后立即验证业务结果，并按约定补齐后续能力。

### Q25：如何处理前后端对接口设计意见不一致？

回答要点：

- 先围绕业务状态、数据所有权和失败语义对齐，不从字段命名争论开始。
- 用具体用户流程和异常案例检验两种方案。
- 对高频变化的展示数据与稳定领域模型做分层，减少彼此耦合。
- 形成可执行的接口契约、错误码、示例和兼容策略。
- 如果仍无法达成一致，明确决策人和期限，避免争论阻塞交付。

### Q26：讲一次你主动发现并解决线上风险的经历。

建议使用 STAR：

- Situation：说明业务背景、影响用户和风险严重度。
- Task：说明自己的职责，以及为什么需要主动介入。
- Action：讲清如何发现、止损、定位、修复、验证和推动协作。
- Result：补充真实的错误率变化、恢复时间或影响范围（待补充）。
- Reflection：说明后来增加了什么监控、测试或流程，避免问题重复发生。

### Q27：如果技术方案最终证明选错了，你会怎么处理？

回答要点：

- 先用事实确认问题来自方案本身，而不是实现、使用方式或外部约束。
- 评估继续修补、局部替换和整体迁移的成本与风险。
- 及时向相关方说明影响、止损方案和决策依据，不隐藏沉没成本。
- 采用兼容层或渐进迁移控制范围，并设置明确的完成标准。
- 复盘当初缺失的信息和验证手段，改进决策过程，而不是追究个人责任。
