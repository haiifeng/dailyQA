### Q2：Card 支付和 Crypto 支付在前端流程上有什么不同？

回答要点：

- Card 支付通常由服务端创建订单或支付会话，前端跳转或嵌入支付组件，最终以服务端回调结果为准。
- Crypto 支付需要处理钱包连接、网络与币种校验、授权、签名、链上交易提交和确认。
- 不能只相信前端“支付成功”；订单状态应由服务端根据支付回调或链上确认更新。
- 需要保证幂等，避免重复点击、重复回调、刷新页面造成重复记账。
- 分清失败、取消、处理中、成功等状态，并允许用户恢复未完成订单。

### Q3：邀请返佣功能如何设计，怎样避免重复返佣？

回答要点：

- 邀请关系在注册或首次绑定时确定，明确能否换绑、是否允许自邀和多级关系。
- 返佣计算放在服务端，前端只展示规则、关系和收益。
- 以业务事件和唯一订单号作为幂等键，退款或订单撤销时需要反向冲正。
- 金额使用最小货币单位或高精度类型，不能直接使用浮点数计算。
- 保存规则版本和流水，保证历史返佣可追踪、可对账。

## 2. React 与 TypeScript

### Q5：React 18 相比早期版本有哪些重要变化？

回答要点：

- `createRoot` 开启新的并发渲染能力，并扩大自动批处理范围。
- `startTransition` / `useTransition` 可把非紧急更新标记为 transition，优先保证输入等紧急交互。
- `useDeferredValue` 适合延迟更新计算或渲染成本高的派生界面。
- `Suspense` 的服务端渲染能力增强，可配合流式输出。
- 并发渲染不是“多线程渲染”，而是渲染工作可中断、恢复和放弃。

### Q6：Hooks 为什么不能写在条件判断中？

回答要点：

- React 依赖 Hook 的调用顺序关联每次渲染中的状态。
- 条件调用会改变顺序，使状态与 Hook 对应错误。
- 条件逻辑应放到 Hook 内部，或拆分为独立组件。

### Q7：如何避免 `useEffect` 引起重复请求、竞态和内存泄漏？

回答要点：

- 正确声明依赖，不通过漏写依赖规避执行。
- 使用 `AbortController` 取消过期请求，或用请求序号忽略旧响应。
- 清理定时器、订阅、事件监听和 WebSocket。
- 开发环境 Strict Mode 可能重复执行副作用，用幂等和 cleanup 验证实现是否安全。
- 服务端数据优先使用成熟的数据请求层统一处理缓存、去重和失效。

### Q8：TypeScript 中 `type` 和 `interface` 如何选择？

回答要点：

- 两者都能描述对象；`interface` 支持声明合并，适合公开、可扩展的对象契约。
- `type` 能表达联合、交叉、条件类型、映射类型和元组，组合能力更强。
- 项目内保持一致比机械地限定某一种更重要。
- 接口返回值应从运行时校验和类型定义两层保证可信，不能只依赖类型断言。

### Q9：如何设计一个可复用的业务组件？

回答要点：

- 先区分展示组件与业务容器，避免组件直接绑定具体接口和页面状态。
- API 保持最小化，用明确的数据和事件传递，不为未知需求预留大量配置。
- 处理 loading、empty、error、disabled 等状态，并考虑键盘操作与可访问性。
- 用组合代替过多布尔属性，必要时提供受控和非受控模式。

## 3. 交易系统与实时数据

### Q10：WebSocket 持续返回大量行情数据时，前端如何处理？

回答要点：

- 接收层与渲染层解耦，数据先写入缓冲区，不要每条消息都触发 React 更新。
- 按固定窗口批量合并；行情类数据可按交易对或主键保留最新值。
- 使用 `requestAnimationFrame` 或节流控制视图刷新频率。
- 大量计算可移到 Web Worker；长列表使用虚拟列表。
- 增加心跳、断线重连、指数退避、消息序号和快照恢复机制。
- 用 React Profiler、Performance 面板和长任务指标定位瓶颈，再决定优化方式。

### Q11：交易页面中服务端状态、本地状态和表单状态怎么划分？

回答要点：

- 行情、订单、持仓等远端数据属于服务端状态，需要缓存、失效、重试与同步。
- 弹窗、选中 Tab、局部交互属于本地 UI 状态。
- 下单输入属于表单状态，需要校验和提交生命周期。
- WebSocket 增量消息应与初始快照合并，并根据版本号或序列号处理乱序。
- 不把所有状态无差别放进一个全局 Store。

### Q12：资产、持仓、当前委托和成交记录如何保持一致？

回答要点：

- 首次进入拉取快照，再消费 WebSocket 增量更新。
- 使用订单 ID、成交 ID 和版本号去重，保证消息处理幂等。
- 下单后可以乐观展示“提交中”，但最终以服务端订单状态为准。
- 断线重连后重新拉快照，避免只依赖断线期间的增量消息。
- 对资金、订单等关键数据建立明确的状态机，避免多个布尔值组合出非法状态。

## 4. Web3

### Q13：钱包连接和链上交易的完整流程是什么？

回答要点：

1. 检测钱包并请求连接，获得账户地址。
2. 校验或切换目标网络，读取余额、Allowance 和业务数据。
3. 必要时先执行 Approve，再发起业务交易。
4. 用户签名后获得交易 Hash，展示 Pending 状态。
5. 等待链上确认并处理成功、失败、替换或长时间 Pending。
6. 刷新链上数据，后端索引存在延迟时给出明确提示。

### Q14：为什么 ERC-20 经常需要 Approve？有什么风险？

回答要点：

- Token 合约通过 Allowance 授权另一个合约代为转账。
- 无限授权体验较好但风险更高；精确授权更安全但可能增加一次交易。
- 前端必须展示授权对象、金额和网络，防止用户在错误网络或恶意合约上签名。
- 金额换算要考虑 Token decimals，避免 JavaScript 浮点精度问题。

## 5. 工程化与浏览器

### Q15：前端项目如何做性能优化？

回答要点：

- 先建立指标：LCP、INP、CLS、首屏资源体积和长任务。
- 加载阶段使用代码分割、按需加载、资源压缩、缓存和 CDN。
- 渲染阶段减少无效更新，合理拆分组件，对真正昂贵的计算使用 memo。
- 大列表虚拟化，高频数据合并更新，动画避免触发布局抖动。
- 图片选择合适格式和尺寸，并配置懒加载。

### Q16：Cookie、localStorage 和 sessionStorage 有什么区别？

回答要点：

- Cookie 会随匹配域和路径的请求发送，可配置 `HttpOnly`、`Secure`、`SameSite`。
- `HttpOnly` Cookie 不能被前端 JavaScript 读取或修改，应由服务端通过响应头设置。
- localStorage 长期保存在浏览器，sessionStorage 通常限定在当前标签页会话。
- 敏感会话信息优先使用安全 Cookie，并同时考虑 CSRF 与 XSS 防护。

## 6. 一面编码题

### 题目 1：合并高频行情更新

实现 `createBatchUpdater(flush, interval)`：连续调用 `push({ symbol, price })` 时，在每个时间窗口结束后按 `symbol` 只保留最新数据，并一次性调用 `flush`。

考察点：Map 去重、定时批处理、资源清理、边界条件。

```ts
type Tick = { symbol: string; price: number };

function createBatchUpdater(
  flush: (ticks: Tick[]) => void,
  interval = 100,
) {
  const pending = new Map<string, Tick>();
  let timer: ReturnType<typeof setTimeout> | undefined;

  function run() {
    timer = undefined;
    if (!pending.size) return;
    const ticks = [...pending.values()];
    pending.clear();
    flush(ticks);
  }

  return {
    push(tick: Tick) {
      pending.set(tick.symbol, tick);
      timer ??= setTimeout(run, interval);
    },
    dispose() {
      if (timer) clearTimeout(timer);
      timer = undefined;
      pending.clear();
    },
  };
}
```

### 题目 2：实现打字机效果 Hook

考察点：Effect 依赖、定时器清理、文本变化时重置、卸载安全。

```tsx
function useTypewriter(text: string, delay = 80) {
  const [value, setValue] = React.useState("");

  React.useEffect(() => {
    let index = 0;
    setValue("");

    const timer = window.setInterval(() => {
      index += 1;
      setValue(text.slice(0, index));
      if (index >= text.length) window.clearInterval(timer);
    }, delay);

    return () => window.clearInterval(timer);
  }, [text, delay]);

  return value;
}
```

## 2. 系统设计

### Q4：设计一个同时支持 Card 和 Crypto 的订阅支付系统。

回答框架：

1. 前端创建统一业务订单，服务端返回订单号和可用支付方式。

6. 支付成功后由服务端开通套餐权益，使用事务或可靠事件保证最终一致。
7. 加入超时查询、补单、退款、对账、监控告警与人工处理入口。
